# Домашнее задание к занятию «Микросервисы: подходы»

## Задача 1: Обеспечить разработку

### Предлагаемое решение: GitLab (Self-managed в облаке) + GitLab CI/CD

Для организации процесса разработки микросервисов предлагаю **GitLab** как единую облачную платформу, которая предоставляет всё необходимое из коробки. Это оптимальный вариант для новичка, так как не требует сложной интеграции разных инструментов.

#### Состав решения:
- **Система контроля версий** – встроенный Git-репозиторий.
- **CI/CD** – встроенный GitLab CI/CD.
- **Хранение секретов** – переменные окружения (CI/CD variables), защищённые и замаскированные.
- **Агенты сборки** – GitLab Runner (можно развернуть на своих серверах или использовать облачные раннеры).

#### Как обеспечиваются требования:

| Требование | Реализация в GitLab |
|------------|---------------------|
| Облачная система | GitLab предоставляет как SaaS (gitlab.com), так и self-managed версию, которую можно развернуть в любом облаке (AWS, GCP, Yandex Cloud и др.). |
| Система контроля версий Git | Встроенный Git-репозиторий с полной поддержкой всех операций. |
| Репозиторий на каждый сервис | Для каждого микросервиса создаётся отдельный проект (репозиторий). |
| Запуск сборки по событию из VCS | Триггеры: push, merge request, tag – настраиваются через `rules` в файле `.gitlab-ci.yml`. |
| Запуск сборки по кнопке с параметрами | Ручной запуск (manual job) с возможностью передачи переменных через UI или API. |
| Привязка настроек к каждой сборке | Каждый job может иметь свой набор переменных, теги, образы. |
| Шаблоны для различных конфигураций сборок | Использование `include` и `extends`, а также `!reference` для переиспользования шаблонов. |
| Безопасное хранение секретов | Переменные CI/CD (защищённые, маскированные, ограниченные по веткам). |
| Несколько конфигураций из одного репозитория | Разные `.gitlab-ci.yml` на ветках или использование `rules` для выбора нужного сценария. |
| Кастомные шаги при сборке | Любые shell-команды, скрипты, вызовы make/docker прямо в секции `script`. |
| Собственные докер-образы для сборки | В параметре `image` можно указать любой публичный или приватный образ (из GitLab Container Registry). |
| Развернуть агентов сборки на своих серверах | GitLab Runner устанавливается на свои серверы, подключается к GitLab и использует их мощности. |
| Параллельный запуск сборок | Настраивается количество параллельных job'ов в раннере и использование матричных сборок (`parallel: matrix`). |
| Параллельный запуск тестов | Разбивка тестов на несколько job'ов (`parallel`), использование `needs` для зависимостей. |

#### Обоснование выбора:
1. **Единый стек** – не нужно собирать пазл из Jenkins + Nexus + Artifactory + Vault. Всё работает сразу после установки.
2. **Простота для новичка** – CI/CD описывается в одном YAML-файле, лежащем в репозитории. Есть готовые шаблоны для большинства языков.
3. **Гибкость** – можно использовать облачных раннеров GitLab (не нужно администрировать), а при росте нагрузки – развернуть свои.
4. **Безопасность** – защищённые переменные, возможность подписывать commit'ы, ролевая модель доступа.
5. **Сообщество и документация** – огромное количество примеров и готовых решений.

Альтернативы (GitHub Actions + Артефакты) тоже хороши, но GitLab даёт более полное out-of-the-box решение для self-managed сценариев.

## Задача 2: Логи

### Предлагаемое решение: EFK Stack (Elasticsearch + Fluentd + Kibana)

Для централизованного сбора и анализа логов в микросервисной архитектуре я выбираю связку **Elasticsearch, Fluentd, Kibana** (EFK). Fluentd заменим на более лёгкий **Fluent Bit** для сбора из stdout, но общая концепция сохраняется.

#### Состав решения:
- **Fluent Bit** – агент на каждом хосте для сбора логов из stdout контейнеров/сервисов.
- **Elasticsearch** – центральное хранилище логов с полнотекстовым поиском.
- **Kibana** – веб-интерфейс для просмотра, фильтрации и создания ссылок на сохранённые поиски.

#### Архитектура:
- Каждый **микросервис** пишет логи в стандартный вывод (`stdout`).

- На каждом хосте запущен **Fluent Bit** (лёгкий агент), который:
  - читает логи из `stdout` всех локальных сервисов/контейнеров,
  - **буферизует** их на диске (для гарантированной доставки),
  - отправляет в центральное хранилище.

- Все логи стекаются в **кластер Elasticsearch**, который:
  - хранит логи в централизованном месте,
  - индексирует их для полнотекстового поиска,
  - обеспечивает масштабирование и отказоустойчивость.

- Разработчики работают через **Kibana** – веб-интерфейс:
  - поиск и фильтрация по любым полям (время, сервис, уровень, текст),
  - сохранённые поиски (можно поделиться ссылкой),
  - дашборды для анализа логов.

**Гарантированная доставка:**  
Fluent Bit поддерживает повторные попытки отправки, запись на диск при потере связи с Elasticsearch, подтверждения (ack). Логи не теряются даже при перезапуске агента или центрального хранилища.

#### Как обеспечиваются требования:

| Требование | Реализация в EFK |
|------------|------------------|
| Сбор логов в центральное хранилище со всех хостов | Fluent Bit на каждом хосте читает логи из docker/containerd или системного journal, отправляет в Elasticsearch. |
| Минимальные требования к приложениям, сбор из stdout | Приложения пишут логи в стандартный вывод (контейнеры делают это по умолчанию). Fluent Bit перехватывает их без изменений кода. |
| Гарантированная доставка логов | Fluent Bit поддерживает буферизацию на диск, повторные попытки и подтверждения (ack). При потере связи данные сохраняются. |
| Поиск и фильтрация по записям | Elasticsearch предоставляет мощный DSL-поиск, Kibana – удобный интерфейс с фильтрами (по времени, полям, тексту). |
| Пользовательский интерфейс для разработчиков | Kibana: можно создавать дашборды, применять фильтры, сохранять поиски. Доступ разграничивается по ролям. |
| Ссылка на сохранённый поиск | В Kibana каждый сохранённый поиск имеет уникальную ссылку, которую можно скопировать и передать коллеге. |

#### Обоснование выбора:
1. **Низкий порог входа** – Fluent Bit имеет простую конфигурацию, Elasticsearch легко поднимается как Docker-контейнер, Kibana даёт понятный UI.
2. **Масштабируемость** – Elasticsearch кластеризуется, выдерживает терабайты логов.
3. **Надёжность** – буферизация на агенте не даёт потерять логи при перезапуске Elasticsearch.
4. **Гибкость парсинга** – Fluent Bit может обогащать логи метаданными (pod_name, service_name) и парсить многострочные логи.
5. **Популярность** – огромное сообщество, множество готовых конфигураций и интеграций.

Альтернативы: Loki + Grafana (проще, но менее мощный поиск), Graylog (тоже хорош, но сложнее в настройке). Для новичка и типовых задач EFK – оптимальный выбор.

## Задача 3: Мониторинг

### Предлагаемое решение: Prometheus + Grafana + Node Exporter + cAdvisor

Для сбора и анализа метрик состояния хостов и сервисов я выбираю **Prometheus** в качестве хранилища метрик, **Grafana** для визуализации, стандартные экспортёры: **Node Exporter** (системные метрики хостов), **cAdvisor** (метрики контейнеров). Для специфичных метрик сервисов – использование клиентских библиотек Prometheus.

#### Состав решения:
- **Prometheus** – сервер сбора метрик (pull-модель) и база временных рядов.
- **Grafana** – дашборды, запросы (PromQL), алертинг.
- **Node Exporter** – сбор метрик ОС (CPU, RAM, HDD, Network) с каждого хоста.
- **cAdvisor** – сбор метрик контейнеров (CPU, RAM, HDD, Network) для каждого сервиса/контейнера.
- **Pushgateway** (опционально) – для кратковременных batch-задач, которые не могут быть затянуты.

#### Архитектура:
- На каждом **хосте** запущены два экспортёра:
  - **Node Exporter** – собирает метрики ресурсов хоста (CPU, RAM, HDD, Network).
  - **cAdvisor** – собирает метрики каждого контейнера/сервиса (потребление CPU, RAM, HDD, Network).

- Каждый **микросервис** (если он написан на Python, Go, Java и т.д.) внедряет клиентскую библиотеку Prometheus и выставляет собственные метрики (например, количество запросов, ошибки, задержки) на эндпоинте `/metrics`.

- Все эти источники метрик (Node Exporter, cAdvisor, `/metrics` сервисов) **периодически опрашиваются (scrape)** сервером **Prometheus**.

- Prometheus хранит метрики как временные ряды и принимает запросы на языке **PromQL**.

- **Grafana** подключается к Prometheus как к источнику данных и позволяет:
  - выполнять произвольные запросы к метрикам,
  - строить дашборды с панелями (графики, таблицы, тепловые карты),
  - настраивать алерты (опционально).

Таким образом, разработчик или администратор через единый веб-интерфейс Grafana может видеть состояние хостов, контейнеров и бизнес-метрик сервисов, а также агрегировать данные (например, средний CPU по всем сервисам).

#### Как обеспечиваются требования:

| Требование | Реализация в Prometheus + Grafana |
|------------|------------------------------------|
| Сбор метрик со всех хостов | Prometheus настроен на `scrape` Node Exporter'а на каждом хосте. |
| Сбор метрик состояния ресурсов хостов: CPU, RAM, HDD, Network | Node Exporter предоставляет все эти метрики. |
| Сбор метрик потребляемых ресурсов для каждого сервиса: CPU, RAM, HDD, Network | cAdvisor собирает метрики для каждого контейнера, сервис можно идентифицировать по лейблам. |
| Сбор метрик, специфичных для каждого сервиса | Разработчики внедряют клиентскую библиотеку Prometheus (например, для Python, Go, Java) и выставляют кастомные метрики (количество запросов, длительность, ошибки). |
| UI с возможностью делать запросы и агрегировать информацию | Grafana: панель "Explore" позволяет писать PromQL-запросы, применять функции агрегации (`sum`, `avg`, `rate` и др.). |
| UI с возможностью настраивать различные панели | Grafana: дашборды создаются перетаскиванием панелей, настраиваются источники данных, переменные, оповещения. Есть готовые дашборды для Node Exporter и cAdvisor. |

#### Обоснование выбора:
1. **Стандарт индустрии** – Prometheus используется в Kubernetes (родной мониторинг), имеет активное сообщество и тысячи готовых экспортёров.
2. **Простота для новичка** – конфигурация описывается в одном YAML-файле (scrape configs). Grafana добавляет интуитивный интерфейс.
3. **Мощный язык запросов** – PromQL позволяет легко строить агрегации, скользящие окна, процентили.
4. **Гибкость** – можно мониторить не только контейнеры, но и виртуальные машины, базы данных, очереди через соответствующие экспортёры.
5. **Масштабируемость** – для больших нагрузок используется Thanos или VictoriaMetrics, но для начала хватит одного Prometheus.
6. **Алертинг** – встроенный Alertmanager или в Grafana.

Альтернативы: Zabbix (тяжеловесный, нет гибких запросов), TICK-стек (InfluxDB, Telegraf, Chronograf) – тоже хорош, но Prometheus стал более распространённым. Для микросервисов, особенно на Kubernetes, Prometheus + Grafana – безальтернативный выбор.

## Итоговая схема взаимодействия всех трёх решений

Поток данных:
Разработчик → GitLab (CI/CD) → сборка образа → регистр → развёртывание на хостах.
С хостов логи собирает Fluent Bit → Elasticsearch → Kibana.
Метрики (ОС – Node Exporter, контейнеров – cAdvisor) → Prometheus → Grafana.
Доступ к Kibana и Grafana у разработчиков и DevOps.

Все три решения интегрируются друг с другом через общие источники данных (хосты, контейнеры) и предоставляют разработчикам единый интерфейс для разработки, отладки по логам и мониторинга состояния системы.
